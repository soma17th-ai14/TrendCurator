# SKILLS.md

이 문서는 TrendCurator 팀이 공유하는 개발 규칙과 아키텍처 기준을 정의합니다.

## 0. AI 에이전트 공통 운영 규칙

이 문서는 Codex, Claude, 기타 AI 개발 에이전트가 공통으로 따라야 하는 팀 기준입니다.

- 모든 문서, 커밋 메시지, PR 제목, PR 설명은 한국어로 작성합니다.
- PR 본문은 `CONTRIBUTING.md`의 템플릿과 예시를 따릅니다.
- 에이전트별 지침 파일인 `AGENTS.md`와 `CLAUDE.md`는 이 문서를 최상위 기준으로 참조합니다.
- 에이전트가 서로 다른 판단을 내릴 경우 `SKILLS.md`의 규칙을 우선합니다.
- 기존 사용자 변경사항을 임의로 되돌리지 않습니다.
- 불확실한 아키텍처 결정은 문서에 가정과 이유를 남긴 뒤 팀 리뷰를 받습니다.

## 1. 프로젝트 아키텍처 원칙

- 모듈 단위로 책임을 분리합니다.
- 개별 스크립트가 아니라 파이프라인 중심 구조로 설계합니다.
- LangGraph 사용을 전제로 상태 기반 처리를 지향합니다.
- 명확한 인터페이스와 데이터 계약을 통해 느슨한 결합을 유지합니다.
- API 엔드포인트, 요청/응답 형식, 공통 에러 형식, 주요 데이터 모델은 `docs/api-spec.md`를 기준으로 맞춥니다.

## 2. 폴더별 책임 규칙

- `app/collectors`: 외부 데이터 수집만 담당합니다.
- `app/agents`: LLM/AI 처리 로직을 담당합니다.
- `app/prompts`: 프롬프트 템플릿과 프롬프트 변경 이력을 관리합니다.
- `app/services`: 파이프라인과 애플리케이션 유스케이스를 orchestration합니다.
- `app/core`: 공통 인프라, 설정, 클라이언트, 공통 유틸리티를 제공합니다.
- `frontend`: 사용자 인터페이스와 통합 화면을 제공합니다.
- `data/profiles`: 수집 소스 프로필과 수집 규칙을 관리합니다.
- `data/samples`: 샘플 문서와 테스트용 예시 데이터를 관리합니다.
- `tests`: 기능 검증을 위한 테스트를 관리합니다.
- `docs`: 아키텍처, 워크플로우, 운영 기준 문서를 관리합니다.

## 3. 코드 작성 규칙

- 함수는 단일 책임을 가져야 합니다.
- I/O와 비즈니스 로직을 분리합니다.
- 환경별 값이나 정책을 하드코딩하지 않습니다.
- 설정은 구현 코드와 분리합니다.
- 구현 전 인터페이스와 데이터 계약을 먼저 확인합니다.
- API, 서비스, 프론트엔드 연동 코드를 작성하거나 수정할 때는 먼저 `docs/api-spec.md`의 계약과 충돌하지 않는지 확인합니다.

## 4. 데이터 구조 규칙

기본 Document 구조는 다음 필드를 포함해야 합니다.

- `title`: 문서 제목 또는 정규화된 헤드라인
- `content`: 본문 내용
- `source`: 원본 URL, 피드, 제공자 이름
- `date`: 발행일 또는 수집일
- `metadata`: 태그, 작성자, 모델명, 주제 라벨 등 추가 구조화 속성

API 레이어에서 노출되는 Document 세부 필드는 `docs/api-spec.md`의 `Document` 모델을 따릅니다.

## 5. 협업 규칙

- 한 파일은 하나의 명확한 역할만 가져야 합니다.
- PR 단위는 작고 리뷰 가능한 크기로 유지합니다.
- PR 본문에는 변경 목적, 변경 내용, 영향 범위, 테스트 결과 또는 테스트 비대상 사유, 문서 변경 여부, 리뷰 포인트, 후속 작업을 포함합니다.
- 구현보다 인터페이스를 먼저 정의합니다.
- 테스트가 필요한 변경은 테스트 없이 병합하지 않습니다. 문서 전용 변경은 PR에 테스트 비대상 사유를 명시합니다.
- 에이전트가 작업할 때는 관련 문서를 먼저 확인하고, 변경 결과가 `README.md`, `docs/`, `SKILLS.md`, `AGENTS.md`, `CLAUDE.md`, `CONTRIBUTING.md`와 충돌하지 않는지 점검합니다.
- 엔드포인트, 응답 스키마, 에러 코드, 모듈별 API 매핑을 바꾸는 작업은 `docs/api-spec.md`를 함께 갱신합니다.
- 임시 파일, 개인 메모, 작업용 spec/plan 등 팀 공유 대상이 아닌 산출물은 `.scratch/` 폴더 아래에 둡니다. 해당 경로는 `.gitignore` 처리되어 레포에 포함되지 않습니다.

## 6. Git 규칙

- `main`은 기본 개발 및 안정 브랜치로 사용합니다.
- 기능 브랜치는 `main`에서 생성합니다.
- 기능 브랜치 이름은 `feature/<short-description>` 형식을 사용합니다.
- 커밋 메시지는 한국어로 작성합니다.
- 커밋 메시지는 `<type>: <요약>` 형식을 권장합니다.
- 권장 커밋 타입은 `init`, `docs`, `feat`, `fix`, `test`, `refactor`, `chore`입니다.
- PR 제목과 설명도 한국어로 작성합니다.
