# 기여 가이드

## 브랜치 전략

- `main`: 안정 릴리스 브랜치
- `feature/<short-description>`: `main`에서 생성하는 기능 브랜치

## PR 방법

1. `main`에서 작은 기능 브랜치를 생성합니다.
2. 변경 범위는 하나의 책임에 집중합니다.
3. 동작, 아키텍처, 워크플로우가 바뀌면 문서를 함께 수정합니다.
4. 동작 변경이 있다면 리뷰 요청 전 테스트를 추가하거나 갱신합니다.
5. `main`을 대상으로 PR을 생성합니다.
6. PR 제목과 설명은 한국어로 작성합니다.

## PR에 반드시 포함해야 하는 내용

PR 본문에는 아래 항목을 반드시 포함합니다.

- 변경 목적: 왜 이 변경이 필요한지 설명합니다.
- 변경 내용: 실제로 무엇을 바꾸었는지 요약합니다.
- 영향 범위: 영향을 받는 폴더, 문서, 기능, 파이프라인을 명시합니다.
- 테스트 결과: 실행한 테스트 또는 테스트 비대상 사유를 작성합니다.
- 문서 변경 여부: README, SKILLS, AGENTS, CLAUDE, docs 변경 여부를 명시합니다.
- 리뷰 포인트: 리뷰어가 집중해서 봐야 할 부분을 적습니다.
- 후속 작업: 이번 PR에 포함하지 않은 다음 작업을 적습니다.

## PR 본문 템플릿

```markdown
## 변경 목적

- 이 PR이 필요한 이유를 작성합니다.

## 변경 내용

- 주요 변경 사항을 항목별로 작성합니다.

## 영향 범위

- 영향을 받는 폴더 또는 기능을 작성합니다.

## 테스트 결과

- 실행한 테스트:
- 테스트 비대상 사유:

## 문서 변경 여부

- 변경한 문서:
- 문서 변경이 필요 없는 이유:

## 리뷰 포인트

- 리뷰어가 확인해야 할 내용을 작성합니다.

## 후속 작업

- 이번 PR 이후 이어서 처리할 작업을 작성합니다.
```

## PR 작성 예시

```markdown
## 변경 목적

- Codex와 Claude를 사용하는 팀원이 동일한 개발 규칙을 따르도록 공통 에이전트 지침을 추가합니다.

## 변경 내용

- `AGENTS.md`를 추가해 Codex 및 범용 AI 에이전트 작업 기준을 정의했습니다.
- `CLAUDE.md`를 추가해 Claude 작업 기준과 문서 우선순위를 정의했습니다.
- `SKILLS.md`에 AI 에이전트 공통 운영 규칙을 추가했습니다.

## 영향 범위

- 문서 변경만 포함됩니다.
- 실행 동작에는 영향이 없습니다.

## 테스트 결과

- 실행한 테스트: 해당 없음
- 테스트 비대상 사유: 문서 전용 변경이므로 실행 테스트 대상이 없습니다.

## 문서 변경 여부

- 변경한 문서: `SKILLS.md`, `AGENTS.md`, `CLAUDE.md`
- 문서 변경이 필요 없는 이유: 해당 없음

## 리뷰 포인트

- Codex와 Claude 지침이 서로 충돌하지 않는지 확인이 필요합니다.
- `SKILLS.md`를 최상위 기준으로 두는 방식이 팀 운영에 적절한지 확인이 필요합니다.

## 후속 작업

- 다음 작업 전 인터페이스 정의 문서를 추가합니다.
```

## 코드 리뷰 기준

- 변경 사항이 폴더별 책임 규칙에 맞는지 확인합니다.
- 인터페이스와 데이터 계약이 명확한지 확인합니다.
- I/O와 로직이 분리되어 있는지 확인합니다.
- 설정값이 하드코딩되어 있지 않은지 확인합니다.
- 동작 변경이 있다면 테스트가 검증하는지 확인합니다.
- 문서 전용 변경이라면 테스트 비대상 사유가 명확한지 확인합니다.
- 필요한 문서가 함께 수정되었는지 확인합니다.

## 커밋 메시지 예시

```text
init: 레포지토리 기본 구조 생성
docs: 워크플로우 문서 추가
docs: 에이전트 공통 작업 지침 추가
feat: 수집기 인터페이스 추가
fix: Digest 날짜 처리 수정
test: 검색 서비스 테스트 추가
chore: 프로젝트 메타데이터 갱신
```
